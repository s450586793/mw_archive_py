## 1. 用户需求记录（原始需求 + 初步思路）
- 需求描述：
  - 当前一次只能归档一个模型；如果某个模型已经在归档，再次发起归档会被阻塞。
  - 需要支持“多模型归档队列”：当已有归档任务执行中时，后续模型进入等待队列，当前任务完成后自动继续处理下一个，直到队列清空。
- 用户初步实现思路：
  - 通过“当前执行中任务 + 待执行列表”的方式，统一串行处理多个模型归档。
- 目标效果：
  - 用户可以连续对多个模型点击归档，无需等待上一个任务完成后再逐个手动操作。
  - 整个归档流程仍保持单任务串行执行，避免目录写入、元数据写入、文件移动等操作发生并发冲突。
- 约束/注意事项：
  - 队列模式是串行执行，不是并发归档。
  - 需要明确单个任务失败时的后续处理策略。
  - 当前阶段先按规范记录需求与分析，不直接开始编码。

## 2. AI 分析（难度 / 思路 / 改动范围）
- 实现难度：中
- 当前代码现状：
  - 后端已有归档核心逻辑，主入口集中在 `app/server.py` 与 `app/archiver.py`。
  - 当前归档更接近“同一时刻只允许一个任务执行”的锁模型，尚未形成“执行中 + 排队中”的任务调度能力。
- 总体实现思路：
  - 在服务端增加归档任务调度器，维护：
    - 当前执行中的任务。
    - 待执行队列。
    - 最近完成/失败结果。
  - 所有“发起归档”请求统一进入调度器，不再直接因为当前繁忙而拒绝。
  - 调度器始终只消费一个任务；当前任务结束后自动拉起下一个任务。
- 推荐实现方案：
  - 保留 `app/archiver.py` 中现有单模型归档能力，作为队列消费者调用的底层执行单元。
  - 在服务端新增队列调度逻辑；如果现有 `server.py` 已较复杂，建议拆到独立模块，例如 `app/archive_queue.py`。
  - 队列中每个任务建议记录：
    - 模型标识。
    - 提交时间。
    - 提交来源。
    - 当前状态：`queued` / `running` / `success` / `failed` / `cancelled`。
    - 错误信息。
  - 建议失败策略为：
    - 单个模型归档失败时记录失败结果。
    - 队列继续处理后续任务，不因单任务失败而整体停摆。
  - 建议后续补充一个队列状态查询接口，用于页面展示“执行中 / 排队中 / 最近结果”。
- 预计改动模块：
  - `app/server.py`
  - `app/archiver.py`
  - 可选新增 `app/archive_queue.py`
  - 可能涉及前端归档按钮与状态展示页面
- 主要风险点：
  - 如果队列只存在内存中，服务重启后未完成任务会丢失。
  - 同一模型被重复提交时，需要定义是否允许重复入队，还是按模型 ID 去重。
  - 前端如果仍把“归档中”理解为全局禁止操作，需要同步调整交互逻辑和文案。
- 验证方案：
  - 连续提交多个模型，验证首个任务立即执行，其余任务进入排队。
  - 验证前一个任务完成后自动处理下一个。
  - 验证其中一个任务失败时，后续任务仍继续执行。
  - 验证接口响应能明确区分“立即执行”与“已入队”。
