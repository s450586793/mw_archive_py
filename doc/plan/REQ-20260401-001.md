## 1. 用户需求记录（原始需求 + 初步思路）
- 需求描述：
  - 当前一次只能归档一个模型；当已有归档任务执行中时，后续归档请求会被阻塞。
  - 需要支持多模型归档排队：当前任务完成后自动处理下一个，直到队列结束。
- 用户初步实现思路：
  - 采用“执行中任务 + 等待队列”的串行消费模型。
- 目标效果：
  - 可以连续提交多个模型归档，不需要等待上一个任务结束再手动点击。
  - 同时保证归档执行仍为单任务串行，避免并发写入冲突。
- 约束/注意事项：
  - 队列是串行执行，不是并发归档。
  - 需要处理同一模型重复提交场景。

## 2. AI 分析（难度 / 思路 / 改动范围）
- 实现难度：中
- 总体实现思路：
  - 把 `/api/archive` 从同步阻塞执行改为“提交任务入队”。
  - 增加后台 worker 线程串行消费队列。
  - 增加任务查询接口，前端通过轮询展示排队/执行/完成状态。
  - 增加同模型 URL 去重：若已在执行或排队，直接复用已有任务。
- 预计改动模块：
  - `app/server.py`
  - `app/templates/config.html`
- 主要风险点：
  - 内存队列在服务重启后不会自动恢复。
  - 前端需要避免重复轮询同一个任务 ID。
- 验证方案：
  - 连续提交多个模型，验证排队和自动串行执行。
  - 重复提交同一模型 URL，验证复用已有任务而不重复入队。
  - 验证任务成功/失败状态和提示文案。

## 3. AI 实现记录（实际落地）
- 实际改动文件：
  - `app/server.py`
  - `app/templates/config.html`
- 关键实现点：
  - 新增归档队列核心结构：`ARCHIVE_QUEUE_PENDING`、`ARCHIVE_QUEUE_TASKS`、`ARCHIVE_QUEUE_HISTORY`、后台 worker。
  - `/api/archive` 改为返回任务对象（`task_id`、状态、队列位置），不再直接阻塞到归档完成。
  - 新增任务状态接口：
    - `GET /api/archive/tasks/{task_id}`
    - `GET /api/archive/queue`
  - 新增同模型去重逻辑：
    - 同一标准化模型 URL 若已在 `running/queued`，返回已有任务并标记 `deduplicated=true`。
  - 配置页归档入口改为“提交任务 + 轮询状态”，支持排队中、执行中、成功、失败提示。
- 与预期差异：
  - 当前队列为内存态，暂未做重启恢复持久化。
- 测试/验证结果：
  - `.\.venv\Scripts\python.exe -m py_compile app\server.py` 通过。
