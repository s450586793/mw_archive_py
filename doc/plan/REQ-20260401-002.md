## 1. 用户需求记录（原始需求 + 初步思路）
- 需求描述：
  - 需要增加一个新的飞书推送渠道。
  - 飞书消息发送链路分为两步：
    - 第一步：通过企业自建应用接口上传封面图。
    - 第二步：通过飞书群机器人 `webhook` 推送消息内容。
  - 页面上需要增加飞书相关配置项，并把配置保存进 `config.json`。
- 用户初步实现思路：
  - 沿用现有通知配置页的结构，新增飞书配置区域。
  - 通过配置填写 `webhook`、`app_id`、`app_secret` 等信息，支持图片上传与消息发送。
- 目标效果：
  - 飞书可作为与 Telegram 并列的可配置通知渠道使用。
  - 推送时先上传封面图，再发送带图片和文本内容的飞书消息。
  - 配置页可直接查看、修改和保存飞书相关参数；服务端重启后仍能从 `config.json` 正常读取。
- 约束/注意事项：
  - 图片上传使用企业自建应用接口，消息发送使用群机器人 `webhook`，两套接口鉴权不能混用。
  - 需要把实现所需的 key、secret、webhook 等配置项写入 `config.json`。
  - 当前阶段先按规范记录需求与分析，不直接开始编码。

## 2. AI 分析（难度 / 思路 / 改动范围）
- 实现难度：中
- 当前代码现状：
  - 现有通知渠道配置位于 `app/config/config.json` 的 `notifications` 节点下，已存在 `telegram` 与 `wecom` 结构。
  - 配置页已有通知设置接口和表单，相关入口在 `app/server.py` 的 `/api/config`、`/api/notify-config`、`/config` 以及 `app/templates/config.html`。
  - Telegram 已形成一套“配置读取 -> 服务注册 -> 页面保存”的通知接入模式，可作为飞书渠道的参考。
- 总体实现思路：
  - 在现有通知抽象层中新增 `feishu` 渠道。
  - 将飞书发送流程拆成两个步骤：
    - 使用企业自建应用凭证获取 `tenant_access_token`。
    - 调用图片上传接口上传封面图，获得可在飞书消息中引用的图片资源标识。
    - 使用群机器人 `webhook` 发送消息卡片或富文本消息。
  - 页面增加飞书配置区，并将配置持久化到 `config.json`。
- 推荐配置结构：
  - 建议把飞书配置放在 `notifications.feishu` 下。
  - 推荐字段：
    - `enable_push`：是否启用飞书推送。
    - `webhook_url`：飞书群机器人 webhook 地址。
    - `app_id`：企业自建应用 `app_id`。
    - `app_secret`：企业自建应用 `app_secret`。
    - `upload_image_enabled`：是否启用封面图上传。
    - `message_template`：可选，预留消息体格式扩展。
    - `web_base_url`：详情页基础访问地址。
- 推荐页面配置项：
  - 启用开关。
  - Webhook 地址输入框。
  - App ID 输入框。
  - App Secret 输入框。
  - 是否上传封面图开关。
  - 详情页基础访问地址输入框。
- 预计改动模块：
  - `app/server.py`
  - 新增 `app/feishu_push.py`
  - `app/templates/config.html`
  - `app/config/config.json`
- 主要风险点：
  - `webhook` 与企业应用凭证分属两套接口，任一配置缺失都会导致推送不完整或失败。
  - 飞书机器人消息体格式存在限制，封面图最终如何在群里展示，需要按可用消息类型验证。
  - 如果配置接口没有对敏感字段做统一处理，后续维护和页面回填可能出现不一致。
- 验证方案：
  - 填写完整配置后，验证封面图上传成功并返回可引用资源。
  - 验证机器人消息成功发送到群中，且包含图片、标题、摘要、详情链接等信息。
  - 验证缺少 `app_id/app_secret`、缺少 `webhook_url`、上传失败等异常场景的报错提示。
  - 验证配置保存后刷新页面和重启服务仍能正确回填。
