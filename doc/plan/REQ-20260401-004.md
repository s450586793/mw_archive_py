## 1. 用户需求记录（原始需求 + 初步思路）
- 需求描述：
  - 当前模型归档支持标题、图片、简介、打印配置、附件等内容，但不支持描述区中的视频链接归档。
  - 希望当 MakerWorld 模型描述里包含 B 站或 YouTube 视频时，本地归档页可以直接嵌入播放，而不是只保留普通文本或空白节点。
- 用户初步实现思路：
  - 已在 	emp/视频链接/ 中保存样例数据，包含：
    - 一个带视频的完整另存页面；
    - 一个 视频链接标签.txt，记录了页面运行态中视频相关 DOM 位置。
  - 先分析 MakerWorld 实际保存的视频结构，再决定归档页应该如何渲染。
- 目标效果：
  - 已归档模型如果描述区包含可识别视频链接，打开本地 index.html 或服务端详情页时，可以直接看到可播放的视频模块。
  - 解析失败或平台暂不支持时，不出现空白占位，至少保留“打开原视频”入口。
  - 老归档模型在执行“重建归档页面”后，也能获得同样的视频展示能力。
- 约束/注意事项：
  - v1 按“在线嵌入播放”实现，不下载视频文件到本地。
  - 优先支持平台：Bilibili、YouTube。
  - 需兼容两种访问方式：
    - 本地后端访问的详情页；
    - 直接打开的离线 index.html 页面。
  - 不走“完整网页另存为播放器资源”方案，避免大体积资源和高维护成本。
  - 仅允许白名单平台的视频嵌入，避免把任意 iframe 直接带入归档页。

## 2. AI 分析（难度 / 思路 / 改动范围）
- 实现难度：中
- 当前代码现状：
  - pp/archiver.py::parse_summary 会从 MakerWorld 数据里读取 summary / summaryHtml 并保存为 meta.summary.html。
  - 样例验证结果显示：MakerWorld 结构化数据里保存的视频节点并不是可直接播放的 <iframe>，而是：
    - <figure class="media"><oembed url="https://www.bilibili.com/video/BV..."></oembed></figure>
  - 	emp/视频链接/视频链接标签.txt 里的 iframe 是页面运行后生成的结果，不是原始归档数据本身。
  - pp/static/js/model.js::renderSummary 原先仅修正简介图片路径，然后直接 innerHTML 注入，不会处理 oembed 或视频链接转换。
  - pp/static/css/model.css 原先没有给简介视频容器准备响应式样式。
- 总体实现思路：
  - 在前端渲染简介 HTML 时，识别 igure.media > oembed[url] 或 data-oembed-url，再把普通视频链接转换成平台对应的嵌入地址。
  - 转换后统一输出响应式 iframe 容器。
  - 若链接可识别但无法构造嵌入地址，则渲染为“原视频链接卡片”而不是空白节点。
  - 若平台不在白名单内，则保持普通链接展示，不做 iframe 嵌入。
  - 该方案不强依赖修改 meta.json 结构，优先复用已有 summary.html 原文，尽量减少归档格式变更。
- 推荐方案：
  - 方案 A：前端渲染时把 oembed 转为 iframe。
- 预计改动模块：
  - pp/static/js/model.js
  - pp/static/css/model.css
  - 	ests/model.video.test.js
- 主要风险点：
  - ile:// 离线页下第三方播放器可能受浏览器、网络、地区策略影响，需准备 fallback link。
  - Bilibili / YouTube 链接形态较多，需要覆盖常见链接格式。
  - 若直接信任任意 iframe，存在注入风险；需限制域名与 URL scheme。
- 验证方案：
  - 使用 Bilibili 样例验证本地详情页可显示视频模块。
  - 使用 YouTube 链接验证能转成 mbed 地址。
  - 对不支持的平台验证展示降级链接而不是空白。
  - 对历史归档执行“重建归档页面”后验证旧 oembed 也能显示视频。
- 可行性结论：
  - 可以实现。
  - 当前缺口不是拿不到视频链接，而是缺少 oembed -> iframe 的前端渲染转换。
