## 1. 用户需求记录（原始需求 + 初步思路）
- 需求描述：
  - 当前模型归档支持标题、图片、简介、打印配置、附件等内容，但不支持描述区中的视频链接归档。
  - 希望当 MakerWorld 模型描述里包含 B 站或 YouTube 视频时，本地归档页可以直接嵌入播放，而不是只保留普通文本或空白节点。
- 用户初步实现思路：
  - 已在 `temp/视频链接/` 中保存样例数据，包含：
    - 一个带视频的完整另存页面；
    - 一个 `视频链接标签.txt`，记录了页面运行态中视频相关 DOM 位置。
  - 先分析 MakerWorld 实际保存的视频结构，再决定归档页应该如何渲染。
- 目标效果：
  - 已归档模型如果描述区包含可识别视频链接，打开本地 `index.html` 或服务端详情页时，可以直接看到可播放的视频模块。
  - 解析失败或平台暂不支持时，不出现空白占位，至少保留“打开原视频”入口。
  - 老归档模型在执行“重建归档页面”后，也能获得同样的视频展示能力。
- 约束/注意事项：
  - v1 按“在线嵌入播放”实现，不下载视频文件到本地。
  - 优先支持平台：Bilibili、YouTube。
  - 需兼容两种访问方式：
    - 本地后端访问的详情页；
    - 直接打开的离线 `index.html` 页面。
  - 不走“完整网页另存为播放器资源”方案，避免大体积资源和高维护成本。
  - 仅允许白名单平台的视频嵌入，避免把任意 iframe 直接带入归档页。

## 2. AI 分析（难度 / 思路 / 改动范围）
- 实现难度：中
- 当前代码现状：
  - `app/archiver.py::parse_summary` 会从 MakerWorld 数据里读取 `summary / summaryHtml` 并保存为 `meta.summary.html`。
  - 样例验证结果显示：MakerWorld 结构化数据里保存的视频节点并不是可直接播放的 `<iframe>`，而是：
    - `<figure class="media"><oembed url="https://www.bilibili.com/video/BV..."></oembed></figure>`
  - `temp/视频链接/视频链接标签.txt` 里的 iframe 是页面运行后生成的结果，不是原始归档数据本身。
  - `app/static/js/model.js::renderSummary` 当前仅修正简介图片路径，然后直接 `innerHTML` 注入，不会处理 `oembed` 或视频链接转换。
  - `app/static/css/model.css` 当前没有给简介视频容器准备响应式样式。
  - `app/server.py::make_summary_payload` 与 `localize_summary_external_images` 只处理文本和图片，不处理视频嵌入。
  - 现有“重建归档页面”能力可用于给历史归档更新新的前端渲染逻辑，无需重新归档原始模型。
- 总体实现思路：
  - 推荐方案：在前端渲染简介 HTML 时，识别 `figure.media > oembed[url]` 或兼容识别 `data-oembed-url`，再把普通视频链接转换成平台对应的嵌入地址。
  - 转换后统一输出响应式 iframe 容器，例如：
    - Bilibili：`https://player.bilibili.com/player.html?...`
    - YouTube：`https://www.youtube.com/embed/...`
  - 若链接可识别但无法构造嵌入地址，则渲染为“原视频链接卡片”而不是空白节点。
  - 若平台不在白名单内，则保持普通链接展示，不做 iframe 嵌入。
  - 该方案不强依赖修改 `meta.json` 结构，优先复用已有 `summary.html` 原文，尽量减少归档格式变更。
- 可选方案对比：
  - 方案 A：前端渲染时把 `oembed` 转为 iframe。
    - 优点：改动集中、兼容历史归档、无需新增 meta 字段、可直接配合“重建页面”生效。
    - 缺点：需要在浏览器端做 URL 识别、白名单校验和失败兜底。
  - 方案 B：归档时把视频链接直接改写成 iframe 并存入 `meta.summary.html`。
    - 优点：归档产物即最终展示结果，前端逻辑更轻。
    - 缺点：历史归档无法自动修复，新增平台时需要重归档或批量改写旧数据。
  - 方案 C：像浏览器“完整网页另存为”一样保存播放器页面及其依赖资源。
    - 优点：最接近页面运行态。
    - 缺点：资源臃肿、平台差异大、维护成本高，不建议采用。
  - 推荐：方案 A。
- 预计改动模块：
  - `app/static/js/model.js`
  - `app/static/css/model.css`
  - `app/archiver.py`
  - `app/server.py`
  - `doc/logs/...`（后续发版时补更新说明）
- 主要风险点：
  - `file://` 离线页下第三方播放器可能受浏览器、网络、地区策略影响，需准备 fallback link。
  - Bilibili / YouTube 链接形态较多，需覆盖：
    - Bilibili `bilibili.com/video/...`
    - Bilibili 短链 `b23.tv/...`
    - YouTube `watch?v=...`
    - YouTube `youtu.be/...`
    - YouTube `shorts/...`
  - 若直接信任任意 iframe，存在注入风险；需限制域名与 URL scheme。
  - 旧归档页只有在重建 `index.html` 后，才能拿到新的前端视频渲染逻辑。
- 验证方案：
  - 使用当前样例中的 Bilibili 链接，确认本地详情页可显示视频模块。
  - 人工补一个 YouTube 链接样例，确认能转成 `embed` 地址并正常渲染。
  - 对不支持的平台或异常链接，确认页面展示为普通打开链接，而不是空白。
  - 对已有历史归档执行“重建归档页面”后，确认旧 `meta.summary.html` 中的 `oembed` 也能显示视频。
  - 校验原有 summary 图片、文本、实例卡片、图库摘要不受影响。
- 可行性结论：
  - 可以实现。
  - 当前“不支持视频归档”的根因不是 MakerWorld 没有提供视频信息，而是现有归档系统只保存了 `oembed url`，但没有把它进一步转换成可播放的嵌入 iframe。
  - 因此这更像是“摘要富文本渲染能力补齐”问题，而不是“重新设计归档架构”问题。
- 建议分阶段实施：
  - 第一阶段：
    - 支持 Bilibili / YouTube 在线嵌入；
    - 增加失败 fallback；
    - 支持重建历史归档页。
  - 第二阶段（可选）：
    - 支持更多视频平台；
    - 在在线编辑器中增加“插入视频链接”能力；
    - 若未来确有需要，再单独评估“视频文件本地化归档”。
