## 1. 用户需求记录（原始需求 + 初步思路）
- 需求描述：
  - 现有 MakerWorld 模型归档流程可以抓取标题、图片、简介、打印配置和 3MF 等内容，但一直没有把模型页面本身的附件一起归档下来。
  - 这些附件通常是组装说明、BOM、补充文档等，用户希望本地归档页也能看到并打开这些文件。
- 用户初步实现思路：
  - 方案 1：先分析 `temp/文档链接/` 中保存的完整页面、DOM 片段、归档后的 `meta.json` 与 `index.html`，确认页面本身是否已经包含附件信息，能否直接从页面归档时下载。
  - 方案 2：如果页面直取不稳定，则利用已有“导入模型时从第一个 3MF 中提取附件”的逻辑，在归档阶段检测模型是否含附件，并在首个 3MF 下载完成后提取附件到归档目录。
- 目标效果：
  - 对于包含页面附件的 MakerWorld 模型，归档完成后本地目录下 `file/` 中应保存对应附件文件。
  - 本地 `meta.json` 需要记录附件结果，归档页能展示附件列表并支持点击打开。
  - 历史归档模型在执行重建或补抓后，也能尽量补齐附件。
- 约束/注意事项：
  - v1 只要求“归档并本地打开附件”，不要求复杂预览。
  - 附件可能来自多个分类，至少要考虑：
    - `Assembly Guide / 组装指南`
    - `Bill of Materials / BOM`
    - `Other / 其他文档`
  - 需要兼容直接打开离线 `index.html` 与本地服务访问两种方式。
  - 避免重复保存同一附件，避免不同来源重复写入造成冲突。

## 2. AI 分析（难度 / 思路 / 改动范围）
- 实现难度：中
- 当前代码现状：
  - `app/archiver.py` 已能从页面 HTML 中提取 `__NEXT_DATA__`，并基于 `props.pageProps.design` 生成 `meta.json`。
  - 目前 `build_meta()` 固定写入：
    - `"offlineFiles": { "attachments": [], "printed": [] }`
  - `temp/文档链接/meta.json` 中也验证了当前归档结果的 `offlineFiles.attachments` 为空。
  - `app/server.py::sync_offline_files_to_meta()` 已经支持扫描模型目录下 `file/` 文件夹，并把附件列表同步回 `meta.json`。
  - `app/server.py` 也已经有现成附件接口：
    - `GET /api/models/{model_dir}/attachments`
    - `POST /api/models/{model_dir}/attachments`
  - `app/three_mf_parser.py` 已具备从 3MF 中提取附件的成熟逻辑，会扫描：
    - `Auxiliaries/Others`
    - `Auxiliaries/Assembly Guide`
    - `Auxiliaries/Bill of Materials`
  - `app/batch_import_service.py` 已复用上述解析结果，把提取出的附件复制到模型目录的 `file/` 中。
- 样例分析结论：
  - 方案 1 不仅可行，而且优先级应高于方案 2。
  - 在 `temp/文档链接/dom.txt` 中，页面运行态 DOM 已明确出现附件区块：
    - `文档 (1)`
    - `组装指南 (1)`
    - 文件名 `维克托美杜莎.pdf`
  - 更关键的是，在保存页的 `__NEXT_DATA__` 中已找到真实结构化字段：
    - `props.pageProps.design.designExtension.design_guide`
    - 样例内容为：
      - `name = 维克托美杜莎.pdf`
      - `url = https://makerworld.bblmw.cn/.../2025-08-27_b875c4323c9538.pdf`
  - 同区域还存在：
    - `design_bom`
    - `design_other`
  - 说明 MakerWorld 页面本身已经提供可直接归档的附件 URL，不需要先下载 3MF 再反向提取，至少在当前样例中如此。
- 总体实现思路：
  - 推荐主方案：页面附件直取。
  - 在归档解析阶段，从 `design.designExtension` 中提取附件清单：
    - `design_guide`
    - `design_bom`
    - `design_other`
  - 将附件统一归一化为结构化列表，例如：
    - `category`
    - `name`
    - `url`
    - `localName`
    - `relPath`
    - `source`
  - 归档时把这些 URL 下载到模型目录 `file/` 下，并调用现有 `sync_offline_files_to_meta()` 同步到 `meta.json`。
  - 同时把结构化附件信息写进 `meta.json`，便于前端按分类展示，而不只是一个纯文件名数组。
  - 如果页面未给出附件 URL，但首个 3MF 已成功下载，则启用兜底方案：
    - 解析首个 3MF
    - 提取 `Auxiliaries/*`
    - 保存到 `file/`
    - 标记来源为 `3mf`
- 可选方案对比：
  - 方案 A：直接从页面 `designExtension` 下载附件。
    - 优点：数据来源明确、无需等待 3MF、附件名与页面展示一致、对非 3MF 附件也有效。
    - 缺点：依赖 MakerWorld 页面结构字段稳定。
  - 方案 B：仅从首个 3MF 提取附件。
    - 优点：可复用现有解析代码。
    - 缺点：只能覆盖被打包进 3MF 的附件；若页面文档与 3MF 内容不一致，会漏抓；还会把“页面附件归档”耦合到“必须先成功下载 3MF”。
  - 方案 C：A 为主，B 为兜底。
    - 优点：覆盖率最高，能兼顾页面直链与历史 3MF 附件提取能力。
    - 缺点：实现稍复杂，需要处理去重与来源标记。
  - 推荐：方案 C，但以 A 为主实现，B 只在页面无附件数据时启用。
- 预计改动模块：
  - `app/archiver.py`
  - `app/server.py`
  - `app/static/js/model.js`
  - `app/static/css/model.css`
  - `doc/logs/...`（发版时补）
- 建议的数据结构：
  - 在保留 `offlineFiles.attachments` 文件名数组的同时，新增 `attachments` 字段保存结构化附件信息，例如：
  - `[{ "category": "guide", "name": "...pdf", "url": "...", "localName": "...pdf", "relPath": "file/...pdf", "source": "page" }]`
  - 这样可兼顾：
    - 现有服务端扫描逻辑；
    - 前端展示分类、原始文件名、来源等信息；
    - 未来做“补抓附件”或“重建页面”。
- 主要风险点：
  - 页面结构字段可能因站点改版发生变动，需要容错：
    - `designExtension` 缺失
    - 分类数组为空
    - 字段名变化
  - 同一附件可能同时出现在页面与 3MF 中，需按 URL、文件名或哈希做去重。
  - 某些附件可能不是 PDF，文件名和 URL 后缀不完全可信，下载命名需要安全化处理。
  - 3MF 兜底若启用，需避免重复解压和重复写入。
  - 历史归档若未保留原始页面 HTML，则无法只靠“重建 index.html”补出页面附件，可能需要补跑归档或基于已下载 3MF 做局部补抓。
- 验证方案：
  - 使用 `temp/文档链接/` 样例模型验证：
    - 归档后 `file/` 中出现 `维克托美杜莎.pdf` 或安全化后的同名文件；
    - `meta.json` 中 `offlineFiles.attachments` 非空；
    - 若新增结构化 `attachments` 字段，其条目包含 `category=guide` 与原始 URL。
  - 页面展示验证：
    - 本地详情页能列出附件并点击下载/打开；
    - 直接打开 `index.html` 时也能访问 `./file/...`。
  - 回归验证：
    - 无附件模型不受影响；
    - 视频、summary 图片、实例 3MF、打印板缩略图逻辑不受影响；
    - 手动上传附件接口仍能正常工作。
- 可行性结论：
  - 可以实现，且优先推荐“页面附件直取”。
  - 当前缺口不是 MakerWorld 页面没有附件信息，而是归档流程尚未读取 `designExtension.design_guide/design_bom/design_other`，也没有把这些文件下载到 `file/` 中。
  - 3MF 附件提取逻辑已经是成熟可复用能力，适合作为页面字段缺失时的兜底，而不是主流程。
- 建议分阶段实施：
  - 第一阶段：
    - 解析页面附件字段；
    - 下载附件到 `file/`；
    - `meta.json` 同步附件结果；
    - 前端展示本地附件列表。
  - 第二阶段：
    - 当页面无附件但已有首个 3MF 时，启用 3MF 提取兜底；
    - 增加历史归档“补抓附件”入口或脚本。
